# 简尚系统 AI 协作与安全开发守则

本文件是硬规则，不是建议。Codex、Claude、Hermes 或其他 AI 接手本项目时，必须先读本文件，再读 `CLAUDE.md` 和 `ALIGNMENT.md`。

## 0. 基本原则

- 系统已经从 MVP 进入试运行前阶段，不能再用“先跑通再说”的方式处理安全、权限和数据。
- 任何功能只要涉及员工、客户、金额、文件、AI 工具、项目状态、权限、部署，都按敏感功能处理。
- 允许小步快跑，不允许无校验、无鉴权、无交接地堆功能。
- 不确定时先停下查代码和交接，不允许凭记忆、凭感觉改。

## 1. 多 AI 协作规则

- 每次只能一个 AI 工作。不得假设 Claude/Codex/Hermes 的改动是最终正确结论。
- 默认岗位分工：
  - Codex：主工程师 / 复杂工程判断负责人。负责主实现、关键技术判断、本地必要联调和疑难排障；只有 UI 专项任务才承担高强度视觉打磨。
  - Claude：执行工程师 / 批量修复与部署负责人。负责已明确方案的实现、重复性修补、文档整理、低成本部署、批量改动和重复验证。
  - Hermes：安全审计官 / 质量稽核官。负责安全审计、权限红队、质量挑刺、上线前风险清单；默认不做大规模功能开发。
- 默认协作模型是“石头、沙子、水”：
  - Codex 是石头：先放主结构、主流程、关键体验和关键技术判断。
  - Claude 是沙子：在主结构之间填补批量遗漏、机械修复、文档收口和部署执行。
  - Hermes 是水：最后做渗透式审查，检查安全、权限、质量和上线风险是否还有缝隙。
- 默认顺序：Codex 主开发 → Claude 修补收口 → Hermes 审计遗漏 → Codex 复核审计结果 → Codex 或 Claude 分批修复。
- 为节省额度，Codex 默认少做大范围扫描；Claude 承担批量机械任务；Hermes 只输出审计，不变成第四个主开发。
- 非 UI 专项任务中，视觉判断主要交给用户人眼；Codex 只做最低限度可用性检查，例如页面能打开、关键按钮能点、没有明显遮挡到无法操作。
- Codex 修复单点问题后，不默认重新全量检查一遍；先按影响面做增量验证，再把需要 Claude/Hermes 复查的内容写清楚。
- Hermes 的输出默认是审计报告和风险清单，不是最终结论。Codex 或 Claude 修复前必须复核问题是否真实、是否属于本轮范围。
- Hermes 不得直接推动大规模重构、批量格式化或跨模块修复；发现系统性问题时先列 P0/P1/P2 和复现依据。
- 开始前必须执行或确认：
  - `git status --short`
  - 阅读 `ALIGNMENT.md` 最新交接
  - 阅读本次相关文件的当前内容
- 不得回滚、覆盖、删除自己不确定来源的改动。
- 不得用 `git reset --hard`、`git checkout --`、大范围格式化、批量删除，除非用户明确要求。
- 工作结束必须写对接：改了什么、验证了什么、没做什么、是否上传服务器。Claude 只在上传服务器后写对接，初审不写。
- 发现其他 AI 的问题时，只修本次目标内的问题；范围外的问题写进交接或代办。

## 2. 权限与数据隔离红线

- 不允许新增无鉴权业务接口。公开接口只能是登录、注册、健康检查、静态资源这类明确安全入口。
- 后端权限不能只靠前端隐藏按钮。所有路由必须在后端校验。
- AI 工具不能绕过业务权限。AI 能不能调用工具、工具能不能读写数据、能读写哪部分数据，是三层判断。
- 所有业务数据读取必须考虑 `role_permissions.data_scope`：
  - `all`：可看全部。
  - `project_related`：只看本人创建、负责、施工参与或被授权项目。
  - `self`：只看本人相关。
  - `private_grant`：只看明确授权资源。
  - `none`：不可读取。
- 私有工作区、总监工作区、上传文件、客户资料，默认不对管理员以外的人开放；若设计为“连超级管理员也不可见”，必须用显式授权表控制。
- Socket、文件下载、AI 工具、导出接口，都必须校验当前用户是否有权访问对应会话、资源或数据。

## 3. 写入、删除、金额与状态

- 新增、编辑、删除、状态推进、金额变更，必须做字段校验。
- 金额必须校验为合法数字；收入/支出类型必须白名单；账户、项目、用户、员工档案必须确认存在。
- 删除必须先确认资源存在，再处理关联数据；不能静默 success 掩盖错误。
- 工程状态只能按流程推进，不允许任意跳状态。
- 高风险动作要有审计或日志：文件上传/下载/删除、AI 写入工具、权限变更、角色变更、金额变更、项目状态推进。
- AI 写入类动作后续必须走后端硬确认，不能只靠 prompt 让模型“记得问一下”。

## 4. 文件与上传

- 业务文件不得直接放 public 暴露路径。下载必须走鉴权接口。
- 部署时不得覆盖线上 `backend/data/`、`.env`、头像上传目录。
- 附件必须记录归属对象：工程、流水、库存、私有工作区或聊天。
- 图片可以预览，文件可以下载，但权限必须一致：看不到业务对象，就不能通过文件中心看到它的附件。
- 文件删除优先软删除；物理清理应单独做任务。

## 5. AI 模块红线

- AI 工具权限不等于业务数据权限。工具允许后，还要继续校验模块权限和数据范围。
- AI 审计/API统计必须记录用户、角色、工具、状态、摘要、耗时、token 或近似成本。
- AI 不能直接返回超出用户权限的数据，即使用户用自然语言绕问。
- AI 不能无确认地创建交易、账户、项目、删除数据、推进工程状态。
- 知识库必须显示真实状态。没有索引、脚本不存在、服务未启动时，要明确报错，不得假装正常。

## 6. 前端体验规则

- 前端页面不能只做“看起来能点”。保存失败、权限失败、网络失败必须给用户明确提示。
- 大表格必须在内部滚动或稳定容器内，不允许撑动整个页面。
- 系统设置、工程详情、财务流水、文件中心等工作页必须稳定，不得切换时抖动、跳位、背景变色。
- 移动端或窄屏不能横向溢出主要内容。
- AI 浮窗、弹窗、抽屉不能遮挡关键输入框，拖动/缩放不能选中底层文字。

## 7. 本地开发与验证

- 默认本地完成，不主动上传服务器，除非用户明确说“上传服务器/上线”。
- 本地开发入口：
  - 前端：`frontend-new/`
  - 后端：`backend/`
  - 启动辅助：`scripts/start-local.command`
  - 真实数据库：`~/fuyulnk/jianshang.db`
- 改后端必须至少跑相关 `node --check`。
- 改前端必须至少跑 `npm run build`。
- 改权限必须用低权限账号或接口逻辑验证一次，不得只用超级管理员看页面。
- 改上传/下载必须验证未登录不可访问、无权限不可访问、有权限可访问。

### 7.1 验证预算与增量检查

- 不要每修一个小问题就重新全量检查。先判断影响面，再选择最小有效验证。
- 已经完成过一次全量检查后，后续小修默认只验证相关文件、相关接口、相关页面或相关权限边界。
- 只有以下情况才重新全量检查：
  - 改了认证、登录态、权限、数据库迁移、状态机、部署脚本、公共工具函数。
  - 修复可能影响多个角色、多个模块、线上数据安全或项目状态流转。
  - 上一次验证发现系统性问题，而不是单点问题。
  - 用户明确要求重新全面审。
- UI 专项任务可以使用浏览器截图和视觉检查；非 UI 专项不要把截图级视觉精修当默认收尾。
- 大任务结束必须写清楚已经验证了什么、没有验证什么、原因是什么；不要用笼统的“已全面检查”代替证据。
- 可交给 Claude / DeepSeek 的重复审计和兼容性检查，不要由 Codex 在同一轮里反复做两遍。

## 8. 部署红线

- 未经用户明确要求，不上传服务器。
- 上传服务器前必须确认本地构建通过。
- 上传服务器必须排除：
  - `.env`
  - `.env.*`
  - `backend/data/`
  - `backend/public/avatars/`
  - `node_modules/`
- 上传前建议备份服务器当前版本。
- 上传后必须验证：
  - `/health`
  - 首页静态资源
  - 登录状态
  - 本次改动涉及的核心接口
- 如果部署后超级管理员权限、数据库、头像、上传文件异常，优先停止继续部署并回查备份和 rsync 排除规则。

## 9. 交付定义

一个任务只有满足以下条件才算完成：

- 代码已按本任务目标实现。
- 没有明显越权、无鉴权、无校验问题。
- 已跑必要检查，并在回复或交接中写明。
- 没跑的检查必须说明原因。
- 没上传服务器时必须明确写“未上传服务器”。
- 遗留问题必须写进 `ALIGNMENT.md`，不能藏在聊天里。

## 10. 遇到危险信号时必须停下

出现以下情况，AI 必须暂停并说明风险：

- 准备覆盖 `.env`、数据库、上传文件、头像目录。
- 准备删除大量文件或重置 git。
- 发现线上和本地数据库结构不一致。
- 发现接口无鉴权、AI 可越权、普通员工能看到不该看的数据。
- 发现部署命令可能清空线上数据。
- 发现自己无法判断某段改动是否来自用户或其他 AI。

本项目可以继续快速迭代，但不能继续无护栏迭代。
