# 简尚系统路线图 V1

> 本文件用于约束 Codex、Claude、Hermes 后续协作。任何新任务开工前先判断：它属于哪条主线，解决哪个真实流程问题，验收标准是什么；不符合主线的功能先暂停，避免拍脑袋扩张。

## 1. 总目标

简尚系统不是做一个通用 CRM，也不是照搬本杰明下单系统。第一阶段目标是把“门店/渠道交接订单 -> 简尚承接施工 -> 材料和人员安排 -> 施工交付 -> 成本结算 -> 完工归档”这条真实流程数字化。

核心判断标准：

- 员工打开系统后，知道自己今天要处理什么。
- 项目工单能按总监真实步骤往下走。
- 每一步有负责人、必填资料、必传附件、确认动作和操作日志。
- 仓库、财务、AI、文件中心都围绕项目工单服务，而不是各自独立存在。
- 系统先服务简尚内部，不做客户登录、商城、经销商体系。

## 2. 当前优先级

### P0 主线：项目工单贴合总监流程

这是当前唯一最高优先级。后续所有功能都要先回答：是否让项目工单更贴合真实施工承接？

目标流程：

1. 门店交接
2. 现场勘察
3. 二次勘察 / 复尺
4. 施工交底
5. 材料出库
6. 进场施工
7. 完工验收
8. 材料回库
9. 班组工费结算
10. 完工成本核算
11. 财务结算
12. 完工归档
13. 售后事件

每一步都要有：

- 状态名称
- 当前负责人
- 可操作角色
- 必填字段
- 必传附件
- 下一步按钮
- 阻塞原因提示
- 操作日志
- AI 可辅助项
- 异常处理方式

### P1 联动线：仓库、财务、文件、AI

围绕项目工单逐步联动：

- 项目工单触发材料出库申请。
- 仓库确认后扣库存，写项目日志。
- 项目完工后触发材料回库 / 损耗记录。
- 班组工费和材料成本进入项目成本核算。
- 项目成本核算结果进入财务结算。
- 所有表格、照片、附件归到项目单据链。
- AI 负责检查缺资料、辅助导入、解释流程，但写入动作必须人工确认。

### P2 体验线：员工能看懂、少点错

目标不是页面更花，而是员工少迷路。

- 每个岗位只看到自己相关的待办。
- 当前页面只突出下一步，不堆所有字段。
- 按钮文案必须像真实工作语言，不用抽象系统词。
- 错误提示必须说明原因和下一步。
- 关键金额、日期、电话、地址要足够醒目。

### 当前阶段边界：先把项目工单板块搞定

当前阶段不要继续扩张手机端、水印相机、外部 MCP、完整 PPT 模板库等旁支。它们都和真实交付有关，但只有在项目工单主流程、资料链节点确认、岗位控制台待办分配跑通后再进入。

本阶段优先完成：

- 项目工单状态机和资料链节点确认动作。
- 每个节点的前置条件、可操作角色、下一步状态和日志。
- 工程、仓库、财务、员工各自控制台能看到对应待办。
- 施工交底单、工勘、复勘、出回库、验收、工费、成本、财务归档在项目详情里能闭环。
- 工勘/复勘的电脑端图片上传、图片说明、PPT 可视图和生成 PPT 必须可靠：图片不能只留 base64 或 UUID 文本，必须能持久化为项目附件并在重新打开弹窗后继续预览。

## 3. 暂停或降级的功能

以下功能保留，但短期不再扩展，避免偏离主线。

### 聊天

现状：已有群聊、附件、图片预览。

短期定位：内部辅助沟通，不是主流程。

暂停内容：

- 私聊增强
- 消息已读未读
- 复杂群管理
- 聊天搜索
- 表情、回复、引用等社交化功能

保留底线：

- 默认群能用
- 文件和图片能发
- 不影响项目工单和文件中心

### 知识库

现状：有知识库状态页和搜索服务口径，但还不是成熟文档库。

短期定位：运维检查入口，不当业务主功能。

暂停内容：

- 大规模知识库 UI
- 文档管理系统
- 自动索引复杂配置
- AI 引用来源美化

保留底线：

- 不误导用户，以“当前只是状态检查”呈现。
- 后续等 SOP 和样板文件稳定后再做正式知识库。

### 个性化设置

现状：可改本地主题颜色。

短期定位：低优先级体验功能。

暂停内容：

- 跨设备同步
- 多主题商城
- 字体/布局深度自定义

保留底线：

- 不影响全局。
- 不破坏页面对比度。

### 控制台大屏 / 花哨统计

现状：有控制台和一些统计。

短期定位：只做真实待办和异常提醒。

暂停内容：

- 看起来厉害但数据不准的指标
- 复杂图表
- 无业务依据的经营分析

保留底线：

- 显示今天/近期要处理的真实事项。
- 数据口径不准时宁愿不显示。

### 文件中心独立扩展

现状：能看最近附件。

短期定位：总目录，不是员工找文件的第一入口。

暂停内容：

- 大文件网盘化
- 文件夹体系
- 批量分享
- 复杂预览器

保留底线：

- 工单附件、流水凭证、仓库单据可查。
- 文件权限跟业务对象一致。

### 财务总览

现状：已有财务页面，但项目成本和结算还没完整打通。

短期定位：预览，不当最终决策。

暂停内容：

- 复杂利润分析
- 自动经营报表
- 多维图表

保留底线：

- 流水录入和附件可靠。
- 后续能承接项目成本核算。

### 手机拍照水印 / 外部水印相机 / PPT 模板库

现状：项目交付资料链已经有工勘图片上传、PPT 生成和首次工勘确认推进的雏形，但还不是完整移动端采集系统。

短期定位：后续阶段能力，不纳入当前项目工单主线收口。

暂停内容：

- 手机 H5 拍照水印页。
- GPS / 时间 / 员工 / 项目水印防篡改。
- 接入今日水印相机等外部平台 API 或封装 MCP。
- 按真实 PPT 模板自动生成三页式工勘 / 复勘 PPT。
- AI 视觉识别现场图片并自动生成问题说明。

保留底线：

- 当前电脑端可以上传现场图片。
- 当前工勘节点可以先人工编辑图片说明并生成 PPT。
- 已上传的工勘图片必须保存为项目附件，单据 JSON 只保留附件引用、说明和顺序；重新打开表格/PPT 可视图时必须能看到图片。
- 图片显示名使用“现场图片 01/02”或人工说明，不能把 UUID/机器文件名直接暴露给员工。
- 当前项目工单主流程优先保证“节点确认 -> 状态推进 -> 岗位待办”。

后续进入条件：

- 项目工单主流程和岗位控制台 V2 已稳定。
- 总监确认工勘 / 复勘 PPT 标准模板。
- 明确水印字段：项目、员工、时间、地址/GPS、照片序号、服务器校验信息。
- 明确是否自研 H5 水印相机，还是采购/接入第三方水印相机平台。

## 4. 短期计划：1-2 周

目标：跑通一条真实施工承接样板单。

### 4.1 项目工单状态机 V2

要做：

- 把当前项目阶段从粗粒度改成总监流程步骤。
- 每一步独立展示当前工作单。
- 每一步只显示当前需要填的字段和上传的附件。
- 未满足条件时，下一步按钮禁用并说明原因。

建议状态：

| 状态键 | 页面文案 | 负责人 | 下一步 |
|---|---|---|---|
| handover_received | 门店交接待核对 | 管理 / 工程 | 现场勘察 |
| survey_pending | 待现场勘察 | 工程 | 勘察完成 |
| survey_done | 勘察完成待复尺 | 工程 | 二次勘察 / 复尺 |
| recheck_done | 复尺完成待交底 | 工程 | 施工交底 |
| briefing_done | 交底完成待出库 | 工程 / 仓库 | 材料出库 |
| material_requested | 已申请出库 | 仓库 | 确认出库 |
| material_out | 已出库待进场 | 工程 / 施工组 | 进场施工 |
| in_progress | 施工中 | 施工组 | 完工验收 |
| inspection_done | 验收完成待回库 | 工程 / 仓库 | 材料回库 |
| material_returned | 回库完成待工费结算 | 财务 / 工程 | 工费结算 |
| labor_settled | 工费结算完成待成本核算 | 财务 | 成本核算 |
| cost_checked | 成本核算完成待财务结算 | 财务 | 财务结算 |
| finance_settled | 财务结算完成待归档 | 管理 / 财务 | 完工归档 |
| archived | 已归档 | 管理 | 售后事件 |

潜在 bug：

- 旧项目状态无法映射到新状态，导致列表空白。
- 前后端状态文案不一致。
- AI 仍使用旧阶段口径。
- 看板统计按旧 phase 计算，数量错误。
- 员工持旧 token，角色变更后权限显示异常。

验收：

- 新建项目后必须从“门店交接待核对”开始。
- 每一步推进后状态、日志、列表标签同步变化。
- 旧项目有兼容映射，不报错。
- 项目列表、详情、AI 查询口径一致。

### 4.2 总监表格字段映射

要做：

- 读取真实样板单，整理每张表的字段。
- 把字段映射到系统字段或附件类型。
- 不急着全部做成结构化表，先标明哪些 V1 结构化、哪些先作为附件。

表格 / 文件：

- 现场勘察表
- 二次勘察 / 复尺表
- 施工交底单
- 材料出库表
- 材料回库表
- 施工班组工费结算单
- 完工成本核算表
- 财务结算凭证

潜在 bug：

- 文件名识别误判，比如“1892/1802”写错。
- 同一个项目多个版本附件，系统不知道哪个是最新。
- 表格字段被合并单元格影响，解析错列。
- 员工上传错项目，附件归属错误。

验收：

- 项目详情能清楚显示每张表“已上传 / 缺失 / 有多个版本”。
- 上传新版本时能看到时间和上传人。
- 不因为附件缺失导致页面报错。

### 4.3 勘察表生成器 V1.1

要做：

- 当前已能生成 HTML 并关联工单，下一步补“确认后推进状态”。
- 选择工单后自动带出客户、地址、电话。
- 勘察完成后提示是否推进到下一步。
- 支持二次勘察 / 复尺类型。

潜在 bug：

- 员工无项目编辑权限，上传成功但回写字段失败。
- 上传附件成功后状态没变，用户以为失败。
- HTML 附件过大超过上传限制。
- 图片太多导致浏览器卡顿。

验收：

- 选择工单 -> 生成勘察表 -> 确认关联 -> 工单详情可见附件和勘察字段。
- 无权限时明确提示“你能生成，但不能写入该工单”。

### 4.4 出库阶段限制

要做：

- 只有交底完成 / 待出库附近状态才能发起材料出库。
- 已归档、售后、未勘察阶段不能申请出库。
- 仓库确认后推进工单状态。

潜在 bug：

- 管理员手动编辑状态绕过限制。
- 同一个项目重复出库申请导致重复扣库存。
- 多仓管同时确认同一申请。

验收：

- 错误阶段发起出库会被拒绝并提示原因。
- 同一出库申请只能确认一次。
- 库存不足不会扣减。

## 5. 中期计划：1-2 个月

目标：让简尚内部各岗位能真的试用。

### 5.1 总监工作区

定位：不改变总监文件夹习惯，把她的工作树搬进系统。

要做：

- 工作树导入器。
- 按项目文件夹识别项目。
- 按文件名识别表格类型。
- 生成待确认清单，不自动入库。
- 支持总监主动授权别人看某个项目或文件。

潜在 bug：

- 文件夹命名不一致导致项目拆错。
- 同名客户多个项目混淆。
- 超级管理员可见性和服务器管理员可见性是两个不同层级，不能混为一谈。
- 批量导入误删或覆盖原文件。

验收：

- 导入一份 3/4/5/6 月样本工作树，系统生成项目清单和附件清单。
- 默认不写入正式项目，必须人工确认。
- 权限边界清楚。

### 5.2 仓库闭环

要做：

- 出库单独立编号。
- 回库单。
- 库存流水。
- 损耗记录。
- 材料成本归集到项目。

潜在 bug：

- 单位换算错误，kg/L/ml/个/份混用。
- 出库和回库跨项目串账。
- 删除产品后历史出库明细丢失名称。
- 库存流水和当前库存对不上。

验收：

- 任意产品能追溯库存变化来源。
- 任意项目能看到材料成本。

### 5.3 财务闭环

要做：

- 班组工费结算。
- 完工成本核算。
- 收支流水关联项目。
- 月度导出表。
- 异常检查：未结算、金额缺失、成本异常。

潜在 bug：

- 流水记了，但没关联项目。
- 项目结算金额和财务流水对不上。
- 批量录入被限流或部分失败。
- 导出表格式不好看，财务不愿用。

验收：

- 一个完工项目能看到：合同金额、材料成本、工费、其他费用、结算金额、利润粗算。
- 月底能导出给财务看的表。

### 5.4 员工工作台

要做：

- 工程：待勘察、待交底、施工中、待验收。
- 仓库：待出库、待回库、低库存。
- 财务：待工费结算、待成本核算、待财务结算。
- 管理：缺资料、异常工单、超期工单。

潜在 bug：

- 待办重复。
- 权限过滤不正确。
- 首页数字和列表数量不一致。

验收：

- 不同角色登录看到不同待办。
- 点待办能直达对应工单和步骤。

### 5.5 AI 内训

要做：

- 更新系统 prompt 到新状态机。
- AI 查询项目时按权限过滤。
- AI 检查缺资料。
- AI 导入工单只生成草稿，不直接创建。
- 高风险写入必须人工确认。

潜在 bug：

- AI 使用旧状态名。
- AI 越权查看其他项目。
- AI 把推断当事实。
- AI 批量创建时日志不完整。

验收：

- 普通员工问无关项目，AI 拒绝。
- AI 能说明某项目下一步缺什么。
- AI 写入前有确认记录和审计。

## 6. 长期计划：3-6 个月

目标：形成简尚自己的经营 / 交付生态。

方向：

- 微信 / 飞书只做通知、导入和提醒，不强迫所有人换沟通工具。
- 移动端优先支持查看待办、上传照片、确认步骤。
- App 壳等系统稳定后再做。
- AI 成为质检员：缺附件、状态异常、成本异常、库存异常、权限异常。
- 建立 SOP 库：门店交接、勘察、交底、出库、施工、验收、结算。
- 建立数据备份、恢复、审计和上线前检查制度。

长期风险：

- 系统越来越复杂，但员工不用。
- 老板只看结果，不理解分阶段交付。
- 数据录入不完整，导致统计无意义。
- 真实流程变动后系统没跟上。
- 多 Agent 并行导致目标分散。

长期验收：

- 新员工能通过系统知道自己岗位流程。
- 总监不改变文件习惯，也能让系统识别和归档。
- 每个完工项目都有完整单据链。
- 管理层能看到真实异常，不是漂亮假数据。

## 7. Agent 分工

### Codex：主开发

负责：

- 核心架构
- 状态机
- 复杂交互
- 视觉体验
- 流程建模
- 风险判断

不负责：

- 频繁重复部署
- 大量机械修小错
- 无主线的功能堆叠

### Claude：补漏和部署

负责：

- 补 Codex 大任务遗漏
- 常规 bug 修复
- 部署上线
- 清理低风险问题
- 对照交接继续推进

注意：

- 不要随意推翻路线图。
- 不要为了看起来完成多而扩展旁支功能。

### Hermes：审计

负责：

- 每天或每轮上线前审查。
- 找 P1 / P2 风险。
- 查权限绕过、数据泄漏、并发、状态不一致、静默错误。
- 不直接主开发，除非明确安排。

审查范围建议：

- 每天晚上常规审查：当天改动 + 高风险路由。
- 上线前审查：权限、数据库迁移、状态流转、库存/金额、附件权限、构建结果。

## 8. 上线前检查清单

每次上线前必须确认：

- `git status --short` 清楚知道有哪些改动。
- `node --check` 覆盖本次改动的后端文件。
- `npm run build` 通过。
- 高风险接口至少做一次本地临时库自测。
- 权限接口未登录返回 401。

### 10.1 仓管出库 + 施工记录 + 完工复勘流水线

**背景：** 施工交底完成后，材料出库、施工过程、完工复勘、材料回库是一条完整的交付链路，当前出库/回库混在一起、施工记录无用、缺少完工复勘。

**改造目标：**

施工交底完成 → 财务/工程确认 → 仓管控制台红色提醒
    ↓
仓管按材料单出库（材料清单/辅材损耗/工具，参照真实 Excel 表格格式+自动计算）
    ↓
确认出库 → 自动推进到"确认进场开工" → 传给工程总监
    ↓
工程总监安排人员、确认进场时间
    ↓
施工过程记录 → 接入水印相机/自建打卡，工人每日拍照签到
    ↓
完工撤场 → 撤场前复勘检查（判断是否返工，防售后投诉）
    ↓
材料回库（完整展示回库明细/剩余/差异）
    ↓
后续步骤不变

**具体的改动点：**

1. **仓管出库页** — 按材料清单/辅材损耗/工具三区展示，仿 Excel 表格格式，有自动计算金额功能（公式计算或 AI 一键计算）。每区一行开始，手动加行。
2. **控制台红色标记** — 有新出库请求时仓管控制台显示红点/红标。
3. **出库面板简化** — 当前"出库明细/回库明细/剩余差异"三个 tab 简化为仅"材料出库"。回库明细放到最后独立板块。
4. **施工记录改造** — 废除单纯填备注，接入水印相机 API 或自建 H5 水印打卡（项目/员工/时间/位置水印），工人每日拍照签到。
5. **完工复勘** — 工人撤场前增加复勘步骤，检查是否需要返工，避免客户投诉再来补。
6. **材料回库独立** — 从出库面板拆出，放到完工复勘后，展示完整回库明细/剩余/差异。
