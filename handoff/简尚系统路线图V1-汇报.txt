简尚系统路线图 V1

说明：
本文件用于向公司内部说明简尚系统当前建设方向、阶段目标、工期预估和技术难点。它不是普通网页，也不是简单复制某个下单系统，而是围绕简尚涂装真实业务流程，搭建一套内部施工承接、仓库材料、财务结算、文件归档和 AI 辅助的经营/交付系统。


一、系统定位

简尚系统第一阶段不做客户商城、不做经销商平台、不做对外销售 CRM。

当前核心目标是：
把“门店/渠道接单 -> 简尚承接施工 -> 工程勘察 -> 施工交底 -> 材料出库 -> 施工过程 -> 验收 -> 材料回库 -> 工费结算 -> 成本核算 -> 财务结算 -> 完工归档”这条内部真实流程系统化。

也就是说，这套系统优先解决公司内部协作问题：

1. 每个项目现在走到哪一步。
2. 哪些资料还没补齐。
3. 哪些材料需要出库。
4. 哪些工地正在施工。
5. 哪些项目待结算。
6. 哪些附件、表格、照片需要归档。
7. 员工、仓库、财务、工程各自该处理什么。
8. AI 如何辅助导入、检查和提醒，而不是乱调用数据。


二、为什么不能按“几天做完一个下单系统”理解

外观看起来只是一个系统页面，但实际要解决的是公司内部流程、权限、数据、附件、库存、财务、AI 调用之间的联动问题。

难点包括：

1. 业务流程不是标准商城流程
   简尚不是普通销售公司，订单多数来自门店/渠道，简尚本质上是施工承接方。系统不能简单照搬商城购物车，也不能照搬客户 CRM，需要按真实施工流程建模。

2. 总监现有流程非常完整
   总监目前本地文件夹里已经有成熟的表格和文件流程，包括现场勘察、二次勘察、施工交底、材料出库、班组工费、完工成本核算等。系统需要尊重现有流程，而不是强行让老员工改变工作方式。

3. 工程项目状态必须严谨
   一个工单不能随便跳步骤。比如没有勘察就不能交底，没有交底不应该出库，没有回库和工费结算就不能完整归档。这里需要后端状态机、前端交互、权限校验同时配合。

4. 权限和数据隔离要求高
   普通员工只能看自己负责或参与的项目，仓库只看材料相关，财务看结算相关，管理员看全局。AI 也必须按权限过滤，不能因为员工乱问就暴露其他项目数据。

5. 仓库库存存在并发风险
   材料出库不是页面显示一下就行，需要保证多个仓管或多人操作时不会重复扣库存、不会出现负库存、不会和项目状态不同步。

6. 文件附件体系复杂
   项目会有大量 Excel、PPT、图片、凭证、勘察表。附件既要能上传下载，又要和工单、财务流水、仓库单据绑定，还要保证权限一致。

7. 表格解析和导入不稳定
   真实 Excel/PPT 文件有合并单元格、命名不统一、版本重复、字段缺失、人工写错字等情况。系统需要先做待确认草稿，不能自动乱入库。

8. AI 不能直接放开
   AI 可以辅助拆单、检查资料、解释流程，但写入项目、创建工单、查财务、查库存等操作必须受权限、审计和人工确认约束。

9. 技术环境不是单一网页
   当前系统涉及前端、后端、数据库、文件上传、服务器部署、PM2 进程、AI 接口、知识库服务、权限审计、构建发布等多个环节，不是一个静态页面。


三、当前技术栈和环境

当前系统主要技术栈：

1. 前端
   Vue 3 + Vite + Element Plus
   用于系统页面、项目工单、仓库库存、财务流水、系统设置、AI 浮窗等。

2. 后端
   Node.js + Fastify
   用于 API 接口、权限校验、项目状态推进、文件上传、AI 工具调用等。

3. 数据库
   SQLite / better-sqlite3
   当前适合内部系统快速迭代，后续如果并发和数据量上来，可以考虑迁移到 PostgreSQL / MySQL。

4. 服务器
   Linux 服务器 + PM2 进程管理
   前端构建后由后端 public 托管，后端通过 PM2 运行。

5. 文件系统
   系统需要保存头像、项目附件、聊天附件、财务凭证、工单文件等。

6. AI 模块
   DeepSeek / OpenAI 兼容接口 + 系统内工具调用
   用于 AI 聊天、工单导入、项目查询、库存查询等。

7. 后续可能涉及
   飞书/微信通知、表格导入、PPT/Excel 模板生成、移动端 App 壳、知识库索引服务。


四、第一阶段计划：2-4 周

第一阶段目标：
先把项目工单做成公司内部可测试的“施工承接主线”，让一个真实项目可以从门店交接一路走到完工归档。

预计周期：2-4 周

这个周期不是只写页面，而是包括：

1. 梳理总监真实流程和文件样本。
2. 把总监每一步映射成系统状态。
3. 改造项目工单状态机。
4. 做每一步的必填字段和附件要求。
5. 做权限和角色控制。
6. 做仓库出库/回库的基础联动。
7. 做项目单据链检查。
8. 做 AI 工单导入和资料检查。
9. 做本地测试、权限测试、构建测试。
10. 做服务器部署和回归检查。

第一阶段要完成的核心步骤：

1. 门店交接
   记录来源门店/渠道、接单人、客户、电话、地址、门店单号、交接备注。

2. 现场勘察
   工程部上传现场图片，生成标准勘察表，并关联到项目工单。

3. 二次勘察 / 复尺
   记录二次复核面积、现场变化、是否具备进场条件。

4. 施工交底
   填写交底日期、施工范围、注意事项、班组确认。

5. 材料出库
   项目工单发起出库申请，仓库确认后扣库存。

6. 进场施工
   记录施工负责人、班组长、施工成员、开工日期、人员状态。

7. 完工验收
   记录完工日期、验收日期、现场问题。

8. 材料回库
   记录剩余材料、损耗、回库状态。

9. 班组工费结算
   录入班组工费，后续进入项目成本。

10. 完工成本核算
   汇总材料成本、工费、其他费用。

11. 财务结算
   记录结算金额、收支流水、凭证附件。

12. 完工归档
   检查资料是否齐全，归档项目。

第一阶段验收标准：

1. 能新建一个项目工单。
2. 工单能按步骤推进，不能乱跳。
3. 每一步缺资料时有明确提示。
4. 工程、仓库、财务、管理员看到不同待办。
5. 出库能扣库存，库存不足不能出库。
6. 勘察表能关联到项目附件。
7. 项目详情能看到单据链缺什么。
8. AI 可以辅助导入工单草稿，但不能绕过人工确认。


五、第二阶段计划：1-2 个月

第二阶段目标：
让公司内部各岗位开始小范围试用，打通仓库、财务和总监工作区。

主要工作：

1. 总监工作区
   按总监现有文件夹习惯做工作树导入，不强迫她改变工作方式。

2. 仓库闭环
   出库单、回库单、库存流水、损耗记录、材料成本归集。

3. 财务闭环
   工费结算、完工成本核算、收支流水、项目利润粗算、月度导出。

4. 员工工作台
   每个岗位只显示自己要做的事，比如工程待勘察、仓库待出库、财务待结算。

5. AI 内训
   让 AI 知道简尚每一步流程，能检查缺资料、提醒下一步、辅助导入。

第二阶段验收标准：

1. 真实项目能完整走完一次。
2. 仓库库存能追溯变化来源。
3. 财务能导出月度汇总表。
4. 总监文件能导入并生成待确认清单。
5. 员工登录后能看到自己的待办。


六、第三阶段计划：3-6 个月

第三阶段目标：
形成简尚自己的内部经营/交付生态。

可能方向：

1. 微信/飞书通知和导入。
2. 移动端拍照上传和现场确认。
3. App 壳封装。
4. AI 自动检查异常项目。
5. SOP 知识库。
6. 备份恢复和上线审计。
7. 数据分析和经营看板。

长期验收标准：

1. 每个完工项目都有完整单据链。
2. 新员工能通过系统知道自己岗位流程。
3. 管理层能看到真实异常，而不是人工问一圈。
4. 总监不用改变原本文件习惯，也能让系统识别和归档。


七、当前已完成的基础

目前系统已经不是从零开始，已有基础包括：

1. 登录和角色权限。
2. 项目工单基础模块。
3. 产品库存模块。
4. 财务账户和流水模块。
5. 文件上传和文件中心。
6. 聊天和群聊基础。
7. AI 浮窗和 AI 工具权限。
8. AI 工单导入助手。
9. 工程部现场勘察表生成器。
10. 项目材料出库联动。
11. 项目单据链汇总。
12. API 审计 / 统计底座。

这些已经证明系统方向可行，但还不能说已经完全适合全员正式使用。下一步重点不是继续堆功能，而是把项目工单主流程打磨完整。


八、当前不优先投入的功能

以下功能不是删除，而是短期不继续扩展：

1. 系统内聊天高级功能
   真实沟通目前仍主要在微信，系统聊天保留基础文件和群聊即可。

2. 个性化主题
   不影响经营交付，暂时不继续扩展。

3. 知识库复杂 UI
   等 SOP 和样板文件稳定后再正式做。

4. 花哨控制台大屏
   数据未完全打通前，复杂图表容易误导。

5. 对外客户系统 / 商城
   当前先做内部施工承接，不做对外平台。


九、主要风险

1. 真实流程没有完全梳理清楚，系统容易做偏。
2. 员工不愿录入资料，导致数据不完整。
3. 附件上传错项目，后续归档混乱。
4. 权限没控制好，可能看到不该看的财务或项目数据。
5. 库存重复扣减或回库不及时，会造成账实不符。
6. AI 如果不加限制，可能误查、误写或越权。
7. 服务器和数据库如果没有备份，出问题恢复成本高。
8. 多个 AI 轮流开发，如果没有路线图，容易偏离主线。


十、开发协作方式

为了降低风险，目前建议按以下方式推进：

1. Codex
   负责主开发、核心流程、复杂交互、架构判断。

2. Claude
   负责补漏、部署、常规修复和大任务后的细节修补。

3. Hermes
   负责审计，重点检查权限、数据隔离、并发、状态一致性和上线风险。

每次开发都应该记录：

1. 做了什么。
2. 改了哪些文件。
3. 怎么验证。
4. 有什么遗留问题。
5. 是否上传服务器。


十一、近期最重要的下一步

下一步不建议继续做旁支功能，应该集中做：

项目工单状态机 V2：按总监真实流程拆步骤。

这一步完成后，系统才会从“有很多模块”变成“一条真实施工流程能跑通”。

建议第一阶段开发周期按 2-4 周预估，因为它涉及：

1. 真实流程梳理。
2. 数据结构调整。
3. 前端交互重构。
4. 后端状态机改造。
5. 权限和角色过滤。
6. 附件和表格映射。
7. 仓库/财务联动预留。
8. AI 流程口径同步。
9. 本地和服务器测试。
10. 旧数据兼容。

如果只做一个“看起来像”的页面可以很快，但如果要让员工真实使用、数据不乱、权限不漏、库存不负、项目不跳步骤，就必须按阶段稳步推进。
